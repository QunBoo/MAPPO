## 差异详细分析

### 差异 1：输出粒度 — 仅图级嵌入 vs 节点级 + 图级嵌入 [🔴 严重]

**论文设计：**

论文第七节明确指出编码器应输出**三类信息**：

- **任务节点嵌入** `h_v`：输入 GRU 解码器，与历史动作、上游决策等结合生成当前决策
- **服务器节点嵌入** `{h_u}`：用于资源分配动作
- **全局图编码** `GraphEnc`：作为解码器的全局条件信息（如初始化 GRU 隐藏状态或注意力上下文）

三者共同支持解码器按拓扑顺序生成每个任务的卸载与资源分配动作。

**项目实现：**

```python
# gnn_encoder.py — GNNEncoder.forward()
dag_emb = self.dag_encoder(dag_x, dag_edge_index)   # (64,)  图级
res_emb = self.res_encoder(res_x, res_edge_index)   # (64,)  图级
joint = torch.cat([dag_emb, res_emb], dim=0)        # (128,)
joint = F.relu(self.joint_proj(joint))               # (128,)
return joint  # 仅返回一个 128 维联合图级嵌入
```

**影响：** 完全丢弃了节点级嵌入，解码器无法获得逐任务的节点信息，GRU 无法按拓扑顺序对每个任务做决策。

---

### 差异 2：上游/下游聚合的融合方式 [🔴 严重]

**论文设计：**

将上游视图嵌入和下游视图嵌入**拼接**，保留方向性信息：

```
h_v^{p+1} = [h_v^(us) ; h_v^(ds)]   # 拼接，维度翻倍
```

上游邻居影响最早开始时间，下游邻居影响关键路径，拼接使解码器可区分两个方向的信息。

**项目实现：**

```python
# gnn_encoder.py — TaskDAGEncoder.forward()
h_fwd = self.conv_fwd1(x, edge_index)
h_bwd = self.conv_bwd1(x, edge_index_rev)
h = h_fwd + h_bwd  # 逐元素求和，维度不变
```

**影响：** 方向性信息被混合丢失。求和操作将上游和下游的信号压缩到同一空间，解码器无法区分信息来源。

---

### 差异 3：服务器图初始特征复杂度 [🟡 中等]

**论文设计：**

```
h_u^0 = { h_u,  f_u^avail }
```

- `h_u`：信道增益矩阵（展平为向量），维度较高，携带丰富的通信状态信息
- `f_u^avail`：可用计算资源（如 CPU 频率）

**项目实现：**

```python
# ResourceGraphEncoder(in_dim=2, ...)
# 注释说明输入为 (4, 2)：[cpu_freq_norm, current_load]
```

**影响：** 仅 2 维标量特征，丢失了大量信道状态信息，服务器节点的通信能力无法被充分编码。

---

### 差异 4：全局图编码缺少 FC 预处理 [🟡 中等]

**论文设计：**

论文第五节要求两步：

1. **全连接层变换**：`ḣ_v = FC(h_v)` — 用于变换维度或特征空间
2. **最大池化**：`GraphEnc = max(ḣ_v)` — 逐维度取最大值

**项目实现：**

```python
# TaskDAGEncoder / ResourceGraphEncoder
graph_emb = h.max(dim=0).values  # 直接 max-pool，无 FC 预处理
```

**影响：** 缺少 FC 变换步骤，节点嵌入直接进入池化，可能无法最优地调整特征空间以突出关键信息。

---

### 差异 5：SAGEConv 内部机制 vs 论文的两步聚合 [🟡 中等]

**论文设计（任务图）：**

两步式聚合：

1. 先对每个邻居用 `W1` 变换 + ReLU → 再均值池化
2. 拼接 `[self; mean]` → 用 `W2` 线性变换 + ReLU

```
h_v'^(us) = ReLU(W1^(us) · h_v'^p)          # 先变换每个邻居
mean = Σ h_v'^(us) / |S_us(v)|              # 再均值池化
h_v^(us) = ReLU(W2^(us) · [h_v^p ; mean])   # 拼接后变换
```

**项目实现：**

`SAGEConv` 默认行为：

1. 拼接 `[self; mean(neighbors)]`
2. 单次线性变换
3. 无中间 ReLU 激活

**影响：** 变换顺序不同（先聚合再变换 vs 先变换再聚合），且缺少中间非线性激活，表达能力有所差异。

---

### 差异 6：任务图输入维度 [🟢 轻微]

**论文设计：** 4 维特征 `[H_{i,j}, C_{i,j}, |Pre_{i,j}|, |Suc_{i,j}|]`

- `H_{i,j}`：任务数据量（MB）
- `C_{i,j}`：任务计算量（Gcycles）
- `|Pre_{i,j}|`：前驱任务数量（入度）
- `|Suc_{i,j}|`：后继任务数量（出度）

**项目实现：** `in_dim=5`，多出一个特征维度，具体含义未在代码中注释说明。

---

### 差异 7：BatchNorm 的使用 [🟢 轻微]

**论文设计：** 未提及任何归一化操作。

**项目实现：** 每层 GNN 后加了 `BatchNorm1d`：

```python
self.bn1 = nn.BatchNorm1d(hidden_dim)
self.bn2 = nn.BatchNorm1d(hidden_dim)
# ...
h = self.bn1(h)
h = F.relu(h)
```

**影响：** 这是常见的工程实践改进，有助于训练稳定性，但改变了特征分布，可能影响与论文的复现一致性。

---

## 编码器修改建议优先级

1. **最高优先级**：修改编码器输出，增加节点级嵌入返回（`h_v`, `{h_u}`），保留图级嵌入作为辅助
2. **最高优先级**：将上游/下游融合从求和改为拼接，并相应调整后续维度
3. **中等优先级**：扩展服务器图特征维度，加入信道增益信息
4. **中等优先级**：在 max-pool 前增加 FC 层变换
5. **中等优先级**：考虑自定义消息传递层以匹配论文的两步聚合机制
6. **低优先级**：明确第 5 维输入特征的含义，或对齐论文 4 维设计
7. **低优先级**：评估是否需要 BatchNorm，若追求严格复现可移除

## 解码器差异详细分析

### 差异 D1：注意力机制完全缺失 [🔴 严重]

**论文设计：**

论文第五节明确描述了三层注意力机制：

1. **注意力分数计算**：`h_{tt'} = w_{v_{i,t}} · h_{v_{i,t'}}`
   - GRU 隐藏状态 `w_t` 作为 **query**，与所有任务节点嵌入 `h_v` 做点积
2. **Softmax 归一化**：`α_{tt'} = exp(h_{tt'}) / Σ exp(h_{tt''})`
3. **上下文向量生成**：`c_{i,t} = [w_{v_{i,t}} ; Σ α_{tt'} h_{v_{i,t'}}]`
   - 将 GRU 隐藏状态与加权求和后的全局任务嵌入**拼接**

注意力机制的核心作用：
- **动态聚焦**：自动关注与当前任务最相关的其他任务
- **全局感知**：克服 GRU 隐藏状态的容量限制，"回顾"整个 DAG 中任意节点
- **结构鲁棒性**：对 DAG 大小和形状不敏感

**项目实现：**

```python
# models/actor.py — Actor.forward()
gru_out, h_next = self.gru(x, h_prev)       # (B, 1, 64)
feat = gru_out.squeeze(1)                     # (B, 64)
offload_logits = self.discrete_head(feat)     # 直接接输出头，无注意力
raw_cont = self.continuous_head(feat)
```

GRU 输出直接接全连接输出头，**完全没有注意力层**。

**影响：** 模型丧失了全局感知能力，仅依赖 GRU 隐藏状态（64 维）传递历史信息，对于长序列或复杂 DAG 结构可能信息不足。这是论文解码器最核心的设计之一，缺失影响很大。

---

### 差异 D2：GRU 输入缺少当前任务节点嵌入 h_v [🔴 严重]

**论文设计：**

论文第四节，GRU 的输入包含 5 个部分：

```
w_{v_{i,t}} = GRU(h_{v_{i,t}}, L^(us)(v_{i,t}), w_{v_{i,t-1}}, a_i^{t-1}, {h_u^t})
```

其中 `h_{v_{i,t}}` 是**当前任务节点的最终嵌入**（来自编码器），是最关键的输入之一，携带了该任务的数据量、计算量、依赖关系等结构化信息。

**项目实现：**

```python
# models/actor.py — Actor._prepare_inputs()
x = torch.cat([gnn_embed, upstream_dec, prev_action, server_embed], dim=-1)  # (B, 160)
```

GRU 的输入由 4 部分拼接而成：
- `gnn_embed` (128)：全局图级嵌入（非当前任务的节点嵌入）
- `upstream_dec` (20)：上游决策
- `prev_action` (8)：前一步动作
- `server_embed` (4)：服务器嵌入

**问题：** 没有传入当前任务 `v_t` 的节点嵌入 `h_{v_t}`。取而代之的是全局图级嵌入 `gnn_embed`，这是所有任务节点的 max-pool 结果，不包含任何特定于当前任务的信息。每个时间步 GRU 看到的图嵌入都相同（同一个 DAG），无法区分不同任务。

**影响：** 解码器无法感知"当前正在为哪个任务做决策"，这是自回归解码器的基本要求。与编码器差异 1（缺失节点级嵌入）关联——因为编码器根本没输出节点嵌入。

---

### 差异 D3：GRU 隐藏状态初始化方式 [🟡 中等]

**论文设计：**

论文第七节和第九节示例指出，全局图编码 `GraphEnc` 应用于**初始化 GRU 隐藏状态**：

```
w_0 = GRU(全局图编码, 零向量)
```

这使解码器在第一个时间步就拥有整个 DAG 的全局概览信息。

**项目实现：**

```python
# models/agent.py — MAPPOAgent
self.h_pi: Optional[torch.Tensor] = None  # (1, 1, 64)

def reset_hidden(self):
    """Reset GRU hidden states at episode start."""
    self.h_pi = None
```

GRU 隐藏状态始终**零初始化**，`GraphEnc` 仅作为 GRU 的普通输入（与 upstream_dec 等拼接），而非用于初始化隐藏状态。

**影响：** GRU 在第一步没有任何先验信息，需要更多时间步来"理解"全局结构。用 GraphEnc 初始化隐藏状态能让解码器立即掌握全局概览。

---

### 差异 D4：服务器嵌入来源与处理方式 [🟡 中等]

**论文设计：**

- 服务器嵌入 `{h_u}` 来自编码器的**服务器节点嵌入**，是 GNN 输出的 64 维向量
- 输入 GRU 时需**聚合为固定维度**（如平均池化）
- 论文公式 54 中 `h_u^t` 带有时间步上标，暗示服务器嵌入可能随时间更新

**项目实现：**

```python
# models/agent.py — MAPPOAgent._slice_obs()
server_embed = obs[..., 25:29]   # 4-dim
```

服务器嵌入直接从观测向量中切片得到 4 维标量，**不来自编码器**。这是原始的观测特征，未经 GNN 编码。

**影响：** 服务器嵌入缺少 GNN 编码后的高维表示，丢失了服务器图拓扑结构信息（如服务器间的通信链路状态）。

---

### 差异 D5：上游决策的处理方式 [🟡 中等]

**论文设计：**

- `L^(us)(v_{i,t})` 是当前任务所有**上游任务的决策集合**，是变长的
- 需要聚合为固定维度（如平均池化）
- 集合内容是上游任务的完整决策（卸载位置 + 资源分配）

**项目实现：**

```python
# models/agent.py — MAPPOAgent._slice_obs()
upstream_dec = obs[..., 5:25]    # 20-dim
```

上游决策从观测中切片为固定的 20 维向量，是**预处理好**的定长表示，而非动态聚合的上游决策集合。

**影响：** 固定维度限制了上游决策的表达能力。当上游任务数量变化时，20 维向量可能不足以编码所有上游决策信息，或包含冗余的零填充。

---

### 差异 D6：动作空间完整性 [🟡 中等]

**论文设计：**

论文第六节明确区分了两种智能体的动作空间：

- **LEO 卫星智能体**：`a_k^t = {z_{n,k}^t, P_n^t, x_{k,t}, f_{k,t}^k}`
  - 带宽分配 z、传输功率 P、卸载决策 x、计算资源分配 f
- **UAV 智能体**：`a_m^t = {z_{n,m}^t, P_n^t, x_{m,t}, f_{m,t}^m, f_{m,t}^k, q_m^t}`
  - 额外包含 UAV 轨迹控制 q 和额外的计算资源分配 f_k

**项目实现：**

```python
# models/actor.py
self.discrete_head = nn.Linear(gru_hidden, 4)   # 卸载到 4 种位置
self.continuous_head = nn.Linear(gru_hidden, 4)  # [bw, comp, dx, dy]
```

所有智能体共享同一个 Actor 网络，动作空间统一为 4 离散 + 4 连续。不区分 LEO 和 UAV 智能体的动作空间差异。

**影响：** UAV 智能体缺少部分动作维度（如额外的计算资源分配 f_k），LEO 智能体包含了不需要的 UAV 轨迹控制维度 (dx, dy)。

---

### 差异 D7：概率分解与链式法则 [🟢 轻微]

**论文设计：**

论文第三节明确给出概率分解公式（公式 53）：

```
π(A_i | G_i, G_net) = Π_{t=1}^{|G_i|} π(a_i^t | L^(us)(v_{i,t}), G_i, G_net)
```

条件依赖 `L^(us)` 确保**因果一致性**：下游任务不会使用未来信息。

**项目实现：**

没有显式的概率分解实现。GRU 通过自回归方式（隐藏状态传递 + prev_action 输入）**隐式地**实现了条件依赖，但条件信息不完全匹配论文定义。

**影响：** 功能上等价，但与论文的理论框架不完全对齐。

---

## 解码器修改建议优先级

1. **最高优先级**：实现注意力机制——在 GRU 输出后增加注意力层，query 为 GRU 隐状态，key/value 为任务节点嵌入，生成上下文向量后再接 Actor 头
2. **最高优先级**：修改 GRU 输入，加入当前任务节点嵌入 `h_{v_t}`（需编码器先输出节点级嵌入）
3. **中等优先级**：用 GraphEnc 初始化 GRU 隐藏状态，而非零初始化
4. **中等优先级**：服务器嵌入改用编码器输出的 GNN 节点嵌入，而非原始观测切片
5. **中等优先级**：上游决策改为动态聚合方式，支持变长上游任务集合
6. **中等优先级**：区分 LEO/UAV 智能体的动作空间，或使用条件动作头
7. **低优先级**：显式实现概率分解，对齐论文理论框架

## 编码器-解码器联合问题

以上编码器和解码器的差异不是孤立的，存在多处**联动关系**：

| 联动问题 | 编码器差异 | 解码器差异 | 影响 |
|----------|-----------|-----------|------|
| 节点嵌入缺失 | 编码器差异 1：仅输出图级嵌入 | 解码器差异 D2：GRU 缺少 h_v 输入 | 解码器无法感知当前任务；注意力无法实现（无 key/value） |
| 服务器嵌入未传递 | 编码器差异 1：服务器节点嵌入被池化 | 解码器差异 D4：使用原始观测而非 GNN 嵌入 | 服务器图拓扑信息完全未传递给解码器 |
| GraphEnc 用途错误 | 编码器差异 4：FC+max-pool | 解码器差异 D3：GraphEnc 未用于初始化 | GraphEnc 的设计初衷是初始化 GRU，而非作为 GRU 的普通输入 |

**核心结论：** 编码器的最大问题（不输出节点级嵌入）直接导致解码器的最大问题（缺少注意力机制和当前任务感知）无法修复。因此修改应从编码器开始，先补齐节点级嵌入输出，再逐步修复解码器。
