# AMAPPO 资源图细粒度建模 Debug 文档与修复计划

## Summary

在 `docs/superpowers/plans` 下新增两份文档：

- `2026-04-27-amappo-resource-graph-debug.md`
- `2026-04-27-amappo-resource-graph-debug-plan.md`

`debug` 文档写清楚当前根因和影响面：
当前项目把资源状态统一压成 4 个聚合节点，根因不只在 `algorithms/amappo_v2.py`，还包括共享环境 `env/sec_env.py`、旧版 `mappo` 构图入口、测试、以及规格文档对 `(4, 2)` / `4` 维 server state 的一致假设。

本次修复按以下目标收敛：

- 资源图节点改为 `R = M + K + 2`
- 节点顺序固定为：`[local, uav_0..uav_{M-1}, sat_0..sat_{K-1}, cloud]`
- `res_x.shape == (R, 2)`
- agent 观测里的 `server_states` 同步扩成逐节点状态
- `amappo_v2` 为主修复路径，`mappo` 同步适配，避免共享环境改动后旧链路失效

## Key Changes

### 1. 环境与资源图接口

- 在 `env/sec_env.py` 中把 `server_loads: (4,)` 重构为逐节点 `resource_loads: (R,)`
- 增加统一资源图构造接口，例如 `get_resource_graph_data()`，返回：
  - `res_x: float32 (R, 2)`
  - `res_edge_index: int64 (2, R*(R-1))`
  - 可选 `node_meta`，用于 debug 文档记录节点类型和原始索引
- 明确资源节点映射规则：
  - `local` -> 全局 local 节点
  - `UAV` -> 当前 agent 对应的 `uav_m`
  - `sat` -> `self._nearest_sat(m)` 对应的卫星节点
  - `cloud` -> 全局 cloud 节点
- `res_x` 的两列统一为 `[current_load, normalized_capacity]`，避免 `load` 与原始 Hz 量级失衡
- `OBS_DIM` 不再写死为 `37`，改为按 `M/K` 推导：
  - `obs_dim = 5 + 20 + resource_node_count + 8`
- `server_states` 在观测中改为逐节点展开，`prev_action` 保持最后 8 维不变

### 2. 训练链路与模型适配

- `algorithms/amappo_v2.py` 和 `algorithms/mappo.py` 不再本地硬编码 4 节点资源图，统一调用环境的资源图接口
- `models/v2/gnn_encoder_v2.py` 与 `models/gnn_encoder.py` 的注释、shape 假设、快速自测样例全部从固定 4 节点改成变量 `R`
- `models/v2/agent_v2.py` 中去掉对固定切片 `29:37` 的依赖：
  - `a_prev` 改为取观测最后 8 维
  - `upstream` 仍保持 `5:25`
  - `server_states` 长度按配置推导
- `Config` 增加或统一派生：
  - `resource_node_count = M + K + 2`
  - `obs_dim = 35 + M + K`
- `Critic` 输入维度继续使用 `config.obs_dim * config.M`，但来源必须是推导值而不是常量

### 3. 文档与规格同步

- `debug` 文档必须包含具体代码定位和影响链：
  - `algorithms/amappo_v2.py` 中 `_build_graph_inputs_v2()` 的 4 节点硬编码
  - `env/sec_env.py` 中 `server_loads`、`resource_graph`、`OBS_DIM=37`
  - `tests/v2/*` 与 `docs/specs/amappo_v2_spec.md` 对 `(4,2)` / `(4,64)` 的固定假设
- 规格同步至少覆盖当前权威文档：
  - `docs/specs/amappo_v2_spec.md`
  - `docs/superpowers/specs/2026-04-22-amappov2-design.md`
- 文档中明确这次不是“只改 `res_x` 一行”，而是环境建模、观测接口、训练入口、测试基线的联动修复

## Test Plan

### A. 依赖与运行前置校验

先补一组环境检查用例，避免“代码改完但当前环境根本不能跑”：

- `python -c "import torch, torch_geometric, pytest"`
- `python -m pytest --version`

当前这台环境已观测到：
- `pytest` 命令不存在
- `python` 缺少 `torch`

所以计划里要把“在 README 约定环境（如 `conda activate appo`）下执行验证”列为前置条件。

### B. 单元测试

新增或改造以下测试，全部按变量 `R` 写，不再写死 `4`：

- 资源图形状测试：
  - `res_x.shape == (M + K + 2, 2)`
  - `res_edge_index.shape[1] == R * (R - 1)`
- 资源更新路由测试：
  - offload 到 `UAV` 时只更新当前 `uav_m`
  - offload 到 `sat` 时只更新最近卫星节点
  - offload 到 `cloud` 时只更新 cloud 节点
- 观测维度测试：
  - `obs.shape[0] == cfg.obs_dim`
  - `server_states` 段长度等于 `resource_node_count`
- GNN 编码器测试：
  - `server_embs.shape == (R, 64)`
  - backward 正常
- Agent 测试：
  - `encode()` 后缓存的 `server_embs` 为 `(R, 64)`
  - `act()` / `evaluate_actions()` 不依赖固定 `obs_dim=37`

### C. 训练集成测试

计划中必须包含“能正常训练”的最小回归用例：

- `AMAPPOv2Trainer._run_episode()` CPU 下可完整执行 1 个 episode
- `MAPPOTrainer._run_episode()` CPU 下可完整执行 1 个 episode
- CLI smoke：
  - `python experiments/train_v2.py --epochs 2 --device cpu --log_interval 1 --save_interval 999999 --mini_batch_size 32`
  - `python experiments/train.py --algo mappo --epochs 2 --device cpu --log_interval 1 --save_interval 999999 --mini_batch_size 32`
- 验收条件：
  - 训练过程无 shape error / index error
  - reward、critic_loss、actor_loss、entropy 全为有限值
  - `global_buffer` 有数据，`_ppo_update()` 可跑通
  - checkpoint 关闭或抬高保存间隔，避免 smoke test 受文件副作用干扰

### D. 慢速健康测试

补 1 个 `slow` 训练健康测试，不要求证明收敛，只验证“修复后训练链路稳定”：

- 运行 5 到 10 个 episode 的 `amappo_v2` CPU smoke
- 断言：
  - 所有指标 finite
  - `critic_loss` 不爆成 `nan/inf`
  - actor loss 非零
  - 资源图节点数与观测维度在整个训练过程中保持一致

## Assumptions And Defaults

- 本次按你的确认执行：资源图包含 `UAV + 卫星 + local + cloud`
- `local` 和 `cloud` 先保留为单节点，不扩成每个 IoTD 一个 local 节点
- `server_states` 同步升级为逐节点状态，因此 `obs_dim` 会变化，不再维持 37
- 主目标是修复当前共享环境下的训练链路，不单独为旧版 `mappo` 保留 4 节点兼容模式
- 当前回合处于 Plan Mode，不写仓库文件；实施时再把上述两份 markdown 落到 `docs/superpowers/plans`
